# ZERDE Philo Blog

Проект философского блога на Next.js (App Router) для платформы студентов КарГУ.

README ниже отражает текущее состояние проекта и все ключевые изменения, которые были реализованы в рамках итераций разработки, багфиксов и адаптации UI к тёмной/светлой теме.

## 1) Технологический стек

- **Фреймворк:** Next.js 15 (App Router)
- **Библиотеки:** React 19 + TypeScript
- **Анимации:** Framer Motion
- **Стилизация:** Tailwind CSS v4, CSS Variables
- **Иконки:** Lucide React
- **База данных/Backend:** Supabase SDK (клиент)
- **Контент:** next-mdx-remote (рендер статей)

## 3) Что было сделано (Последние обновления и фиксы)

### 🎨 Оформление, Типографика и CSS
- **Шрифты:** Восстановлен правильный шрифт для основного текста (`Source Serif 4`). Главный акцентный шрифт (`ZerdeDisplay`) теперь изолирован и применяется строго к логотипу и основным навигационным ссылкам первой строки Header-а. Для всплывающих категорий применен стандартный системный шрифт/UI-шрифт.
- **Тёмная / Светлая тема (Dark Mode):** 
  - Исправлен баг "чёрного фона", когда сайт непроизвольно адаптировался под системные настройки ОС.
  - В `ThemeProvider.tsx` отключена автоматическая проверка `(prefers-color-scheme: dark)`, стартовая тема теперь контролируется предсказуемо.
  - Адаптирована строка **категорий в Header**: теперь выпадающее меню динамически подстраивает свой фон (`var(--color-bg)`) и цвет текста (`var(--color-text)`, `var(--color-text-muted)`) под текущую тему без хардкода `#ffffff` и `#111111`.
  - Эффект **"Шторки" перехода (`PageCurtain.tsx`)** переделан: вместо инвертированного текста он теперь использует фоновый цвет текущей темы (`var(--color-bg)`).

### 🧩 Компоненты и Навигация
- **Глобальный Header:** Избавлен от багов маршрутизации и стилей, убраны задвоенные категории. Цвета `active` состояния работают исправно в обеих темах: светлой и тёмной.
- **Удаление дубликатов табов:** Вырезан локальный компонент `<CategoryTabs />` с индивидуальных страниц (`/posts`, `/forum`, `/news`, `/materials`, `/podcasts`), так как навигация по категориям уже встроена в глобальный Header (чтобы избежать "двойного" хидера на странице).
- **Page Transitions:** Стабилизирована шторка (`PageCurtain.tsx` / `PageTransition.tsx`), переходы теперь не вызывают ложных срабатываний анимации с неправильным фоном.

### ⚙️ Инфраструктура и Стабилизация
- **Кэширование Next.js:** В процессе отладки были устранены ошибки маршрутизации и `hydration mismatch`, возникающие из-за старого `.next` кэша.
- **Dev-сервер:** Исправлены проблемы с зависанием портов (очистка порта 3000 при перезапуске).

## 5) Инструкция по запуску

Из корня проекта `philo-blog`:

```bash
# 1. Установка зависимостей
npm install

# 2. Запуск локального dev-сервера (порт 3000)
npm run dev
```

Сборка для production:

```bash
npm run build
npm run start
```

## 6) Переменные окружения (.env.local)

Для работы с Supabase требуются следующие ключи:

```env
NEXT_PUBLIC_SUPABASE_URL=your_supabase_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key
```

*(Если ключи отсутствуют или стоят заглушки — браузерный клиент отключает функционал работы с БД корректно, без падения приложения).*

## 9) Почему "медленно" именно сейчас

Если запускать через `npm run dev`, медленный первый вход в `/admin` складывается из:

- первой компиляции админ-маршрута в dev;
- проверки авторизации и роли через middleware (JWT-first, fallback в БД);
- инициализации клиентских интерактивных компонентов админки.

В production (`npm run build && npm run start`) первая загрузка обычно заметно быстрее и стабильнее.

### 11.2 Админка: создание, редактирование, удаление

  - открыть запись;

### 11.3 Форма статьи (article workflow)

- Убран старый боковой блок публикации/категорий для article-режима.
- Для article включен publish-only режим (без черновика).
- Категория статьи назначается автоматически (по категории "мақала/maqalalar").
- Добавлена загрузка файла статьи (PDF/DOCX) в Supabase Storage.
- Добавлен блок источника (source URL) в сайдбар статьи.
- Добавлена загрузка обложки статьи с локальным превью и последующей загрузкой в Storage.
- Добавлено поле `АВТОР АТЫ` (кастомное имя автора статьи).
- Добавлен большой текстовый редактор статьи (`МАҚАЛА МӘТІНІ`) с живым счетчиком символов.

### 11.4 Slug: полная автоматизация

- Ручной ввод slug убран из рабочего процесса редактора.
- Slug теперь генерируется автоматически из заголовка.
- Добавлена транслитерация кириллицы (рус/каз) в латиницу.
- Добавлена авто-уникализация slug при сохранении (суффиксы `-2`, `-3` и т.д.).

### 11.5 Страница статьи `/posts/[slug]`

- Устранен runtime-крэш от `next/image` при внешних URL, не входящих в whitelist.
- Добавлена безопасная проверка `cover_url` (рендер обложки только для image URL).
- Отображение контента статьи улучшено:
  - нормализуется перенос строк;
  - длинный неразмеченный текст автоматически делится на читаемые абзацы.
- В шапке статьи добавлено `Время чтения X мин`.
- Кнопка скачивания статьи перенесена вверх (сразу под мета-информацией).
- Удален нижний блок автора (карточка вида "Al-Farabi Tusup"), чтобы не перегружать страницу.
- Уменьшен размер заголовка статьи и улучшен ритм текста для более комфортного чтения.

### 11.6 Типы, запросы и модель данных

- Добавлено поле `file_url` в типы и маппинг постов (для статьи/материала).
- Добавлено поле `author_name`:
  - в `lib/supabase/types.ts`;
  - в `types/blog.ts`;
  - в `lib/queries/posts.ts` (select + map).
- На странице статьи автор отображается по приоритету:
  - `author_name`;
  - профильный `full_name`;
  - fallback.

### 11.7 Supabase Storage и RLS

- Добавлен SQL-файл `supabase/storage-policies.sql` с инструкцией по ручной настройке.
- В файл включены:
  - создание публичного bucket `media` (idempotent);
  - Storage policies на insert/update/delete/select;
  - policies для `public.posts` на insert/update/delete для authenticated users;
  - миграция колонки `author_name`:
    - `ALTER TABLE public.posts ADD COLUMN IF NOT EXISTS author_name text;`
- Upload path в Storage приведен к user-scoped формату (`folder/userId/timestamp.ext`) для совместимости с RLS.

### 11.8 Прогон и валидация

- После каждой крупной серии изменений выполнялась сборка `npm run build`.
- На текущем этапе сборка проходит успешно.
- Оставшиеся предупреждения в build касаются только `img` в подкаст-страницах и не блокируют работу.

## 12) Что нужно сделать вручную после pull

1. Выполнить SQL из `supabase/storage-policies.sql` в Supabase SQL Editor.
2. Убедиться, что bucket `media` существует и имеет `Public = true`.
3. Проверить, что у редактора/админа корректная аутентификация (для RLS insert/update/delete).
4. Проверить e2e-флоу:
   - создание article с текстом;
   - создание article с файлом;
   - редактирование article;
   - удаление post из `/admin/posts`.
